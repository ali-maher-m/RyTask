# RyTask docs

The documentation site for [RyTask](https://rytask.app), built with
[Fumadocs](https://fumadocs.dev) (fumadocs-core + fumadocs-ui + fumadocs-mdx + fumadocs-openapi)
on Next.js App Router. Content is organized by the [Diátaxis](https://diataxis.fr) model:
Tutorials, How-to guides, Reference, Explanation.

## Run it

From the repo root:

```bash
pnpm install
pnpm --filter @rytask/docs dev     # http://localhost:3002
```

`dev` and `build` first regenerate the MCP tool pages (see below), then run Next.

## Build

```bash
pnpm --filter @rytask/docs build   # or `pnpm build` at the root (Turborepo builds everything)
```

The build is static-friendly: every docs page is prerendered, search runs locally (Orama), and
`/llms.txt` + `/llms-full.txt` are emitted for AI agents.

## Generated reference pages

Two parts of the Reference section are generated, not hand-written:

- **REST API** (`/docs/reference/rest-api`) — rendered by `fumadocs-openapi` from three OpenAPI
  documents referenced in `lib/openapi.ts`: the committed M0/M1 contract specs
  (`specs/002-identity-tenancy-onboarding/contracts/openapi.yaml`,
  `specs/001-core-work-loop/contracts/openapi.yaml`) and `openapi/time-slack-mcp.yaml`, which
  documents the M2/M3 surface and is maintained here, verified against the controllers in
  `apps/api`. The pages are virtual — nothing to regenerate; edit the YAML and rebuild.
- **MCP tools** (`/docs/reference/mcp-tools`) — one page per tool, generated by
  `scripts/generate-mcp-pages.mjs` from the live registry in `@rytask/contracts` (names,
  descriptions, zod input schemas) plus the permission map in
  `apps/api/src/mcp/tools/tool-dispatch.ts`. Regenerate with:

  ```bash
  pnpm --filter @rytask/docs generate:mcp
  ```

  `scripts/check-mcp-docs-drift.mjs` runs on every build and fails it if the registry and the
  generated pages disagree (e.g. a 50th tool lands without regenerating).

Generated tool pages are gitignored; they are rebuilt on every `dev`/`build`.

## Theming

All colors, type, spacing, radii, and motion come from the RyTask design system. The single
source of truth is `branding/colors_and_type.css`, imported directly (not copied) by
`app/global.css`, which then maps Fumadocs' theme variables onto the semantic `var(--*)` tokens.
Light and dark mode resolve from the same token names — next-themes sets both the `dark` class
(for Fumadocs internals) and `data-theme` (for the RyTask tokens). Fonts are Hanken Grotesk
(UI), Schibsted Grotesk (brand moments), and Geist Mono with `tabular-nums` for figures, loaded
via `next/font`. Logo assets in `public/` are copies of `branding/assets/`.

When adding styles, reference semantic tokens only — never raw hex values, gradients, or
blurs (the brand is flat and token-only).

## Writing content

- Pages live in `content/docs/<section>/…` as MDX; sidebar order comes from each folder's
  `meta.json`.
- Frontmatter is `title` + `description` only.
- Status badges: `<StatusBadge status="available|in-progress|coming-soon" />` inline, and the
  `<ComingSoon tier="v2" />` / `<InProgress />` banners for unshipped features. These are
  available in every MDX page without imports (`components/mdx.tsx`).
- Voice: plain, kind, sentence case, no emoji. Never document behavior you haven't verified in
  `apps/api` / `apps/web` — for planned features, say so with a badge.

## Deploy

The app is a standard Next.js site and deploys anywhere Next runs:

- **Vercel**: set the project root to `apps/docs`; build command
  `pnpm --filter @rytask/docs build`; install command `pnpm install` at the repo root.
- **Cloudflare Pages / others**: same build command; serve with
  `pnpm --filter @rytask/docs start` (port 3002) or adapt to the platform's Next.js preset.

The site needs no environment variables and no backend — search and llms routes are computed
from the content at build time.
