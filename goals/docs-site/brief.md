# RyTask documentation site — full brief

Build the complete RyTask documentation site in this monorepo using **Fumadocs** and the
**Diátaxis** content model. This file is the authoritative brief; the `/goal` condition points here.

## Project context — read this first, it corrects CLAUDE.md

CLAUDE.md says this is a "planning repo, pre-code". That is **outdated**. The codebase is real and large:

- Milestones M0, M1 (core API: auth, orgs, workspaces, projects, work-items), 003 (full Next.js
  frontend), M2/005 (native time tracking, the flagship), and M3/004 (Slack capture + the 49-tool
  MCP server) are **complete and committed on main**.
- M4 (006-m4-reporting: the "Where did my time go?" report, interruption ledger, My week) is
  **in progress** on the current branch with **uncommitted changes** in the working tree.
- Verify implementation status yourself from the code, `git log`, and `specs/` — never document a
  feature as shipped without confirming it exists in `apps/api` or `apps/web`.

Do **not** read `knowledge/BUILD-PLAYBOOK.md` (per project rules).

## Hard safety constraints (non-negotiable)

- Do **not** switch branches, commit, stage, stash, revert, or clean ANYTHING. The working tree
  holds in-progress M4 work that is not yours. The user will commit.
- Only **create** new files under `apps/docs/` (plus, only if strictly required, minimal workspace
  wiring: root lockfile via `pnpm install`, a `turbo.json` task entry). Do not modify `apps/api`,
  `apps/web`, `packages/*`, `specs/`, `knowledge/`, or `branding/` source.
- Zero changes to existing CI gates' behavior. `apps/docs` must not break `check-required-tests`,
  `check-mcp-parity`, or boundary checks.
- Make the site deployable to Vercel/Cloudflare Pages, but do **not** deploy anywhere.

## What to build

A complete documentation site for RyTask (rytask.app) — an open-source, AGPL, self-hostable
project-management & issue tracker. The audience is developers and companies who fork/clone the
repo and run it on their own servers, plus their non-technical end users (docs prose must pass the
project's "Albert/Marissa" plain-language test: sentence case, jargon-free, kind).

The docs must explain **everything**: every feature, every integration, every config knob, every
API surface — implemented ones in full depth, planned ones (v2/v3 tiers in `knowledge/features.md`,
and anything in `specs/` not yet merged) as "Coming soon" pages so readers see the full roadmap.

Stack: `apps/docs` built with Fumadocs (fumadocs-core + fumadocs-ui + fumadocs-mdx, latest stable)
on Next.js App Router, in the existing pnpm + Turborepo workspace, linted by Biome.

## Information architecture (Diátaxis)

### 1. Tutorials (learning-oriented)

- **Quickstart**: self-host with docker compose → create org → first workspace/project → first
  task → log time, in ~15 minutes. Use the REAL compose file in `infra/docker` and the real env
  vars (note the known gotchas: `JWT_SECRET` is required; check git history for committed-infra
  defect fixes).
- **First week with RyTask**: capture from Slack, track time against an estimate, read the
  plan-vs-actual meter, connect an AI agent via MCP.

### 2. How-to guides (task-oriented)

- **Self-hosting & administration**: requirements, docker compose production setup, every
  environment variable, reverse proxy/TLS, backups & restore, upgrades & migrations
  (`drizzle-kit migrate`, never `db:push`), object storage (MinIO/S3), email, scaling api vs
  worker (same image, `WORKER=1`).
- **User guide, one page per feature area**: organizations & members, workspaces, projects, work
  items (statuses, labels, priorities, trash/restore), fast capture syntax
  (`@assignee ~estimate #label`), time tracking (timers, manual logs, estimates, plan-vs-actual
  meter, over-budget behavior), reporting (mark per current M4 status), inbox/notifications,
  permissions & roles.
- **Slack integration**: install the bot, slash command, capture modal, user mapping, disconnect,
  troubleshooting.
- **MCP server**: what it is, connecting Claude/Claude Code/other agents (HTTP + stdio
  transports), agent access tokens UI, the 100%-parity guarantee, security model.
- **GitHub App and any other v2/v3 integrations**: "Coming soon" pages.

### 3. Reference (information-oriented; generated where possible)

- **REST API**: generated from the `packages/contracts` OpenAPI spec via `fumadocs-openapi`.
- **MCP tools**: one page per tool for ALL 49, generated by a script (write it, e.g. in
  `apps/docs/scripts/`) that reads the MCP tool schemas from `packages/contracts` and emits MDX —
  include params, descriptions, permissions. Add a small drift check (script exits non-zero if
  tool count ≠ docs pages) wired into the docs build.
- **Environment variables table**; **permissions/roles matrix**; **capture-syntax grammar**;
  webhooks/idempotency behavior; rate limits if any (verify in code).
- **Feature-status matrix**: every capability from `knowledge/features.md` with a status badge
  (Available / In progress / Coming soon) and tier (MVP / v2 / v3).

### 4. Explanation (understanding-oriented)

Distill from `knowledge/ARCHITECTURE.md`, `VISION.md`, `features.md` — rewrite for outsiders,
don't paste internal docs:

- **Why RyTask**: positioning, the nine differentiators D1–D9, CTW north-star.
- **Architecture overview**: modular monolith, module boundaries, api+worker single image,
  event-driven, ports & adapters.
- **Multi-tenancy & data isolation**: organizationId everywhere, TenantGuard, tested isolation.
- **The time-tracking model**: plan vs actual, interruption philosophy.
- **MCP parity philosophy**: API-first, agents as first-class clients.
- **Testing & quality gates**: why integration tests run on real Postgres.
- **Contributing**: monorepo layout, Spec-Driven Development workflow (Spec Kit), how to add a
  module, license (AGPL).

## Status marking rules

- Source of truth for the feature universe: `knowledge/features.md` capability matrix + tiers;
  `knowledge/REQUIREMENTS.md` for behavior detail; `specs/<feature>/spec.md` for shipped feature
  behavior; the actual code for ground truth.
- **Implemented** (M0–M3, 003, 005 on main): full documentation with real behavior, verified
  against code.
- **M4 reporting** (006, in progress, uncommitted): document as "In progress / coming in the next
  release" with the planned behavior from `specs/006-m4-reporting/spec.md`.
- **v2/v3 tier features**: a real page each with a one-to-three-paragraph description and a
  prominent "Coming soon" badge. Never fake screenshots or invent behavior for unshipped features.
- Build the badge as a reusable MDX component colored with semantic tokens only.

## Brand & writing rules

- Theme tokens come from `branding/colors_and_type.css` — wire them in as the upstream source
  (import/reference, don't copy hex). Sunbeam yellow primary always carries dark ink text; Honey
  accent for time/momentum; warm Stone neutrals; light AND dark mode from the same semantic names.
- Hanken Grotesk for UI/body, Schibsted Grotesk 800 only for brand moments, Geist Mono
  `tabular-nums` for every figure, ID, and inline value.
- Flat fills, no gradients, no glassmorphism, small radii, 1px hairlines, calm motion,
  `prefers-reduced-motion` respected. No emoji as UI chrome.
- Voice: plain, kind, jargon-free, sentence case. Every page must be understandable by a
  non-technical teammate where the topic allows.
- Biome style: single quotes, 2-space, 100 cols, trailing commas, LF.

## Required platform features

- Built-in local search (Orama) working.
- `llms.txt` and `llms-full.txt` routes exposed (AI-agent-readable docs — this is an MCP-first
  product; agents will consume the docs).
- Light + dark mode from the same semantic token names.
- A README in `apps/docs` explaining how to run, build, regenerate reference pages, and deploy.
