## What this skill does

Brings `apps/docs` (the Fumadocs site served at docs.rytask.app) in line with the code. The docs serve two audiences — self-hosters and end users — and they promise to be "verified against the code", so every claim you write must be checked against the actual source, not inferred from commit messages.

## Step 1 — Establish what changed

Work out the set of code changes the docs must absorb:

- If the user gave a commit range, branch, or PR: `git diff --stat <range>` and read the diff.
- If the user described the change in words: find the actual code (the change may already be merged) and read it. Never document a described behavior without confirming the code really does it.
- If neither: default to what the current branch adds over `main` (`git diff main...HEAD --stat`) plus uncommitted changes (`git status`, `git diff --stat`). If you're on `main` with a clean tree, look at recent commits (`git log --oneline -15`) and confirm the intended range with the user before doing a large sweep.

From the changed paths, build a list of *user-visible effects*: new/changed env vars, new/removed MCP tools or changed tool descriptions/schemas, new REST endpoints, changed UI behavior, changed Slack capture syntax, changed Docker/compose setup, changed permissions. Internal refactors with no visible effect need no doc changes — say so and stop.

## Step 2 — Map code areas to doc pages

All content lives in `apps/docs/content/docs/`. Use this mapping to find every page a change touches (a single change often touches several):

| Code area changed | Doc location | How it's maintained |
|---|---|---|
| `packages/contracts/src/mcp/registry.ts`, `tool-io.ts`, `apps/api/src/mcp/tools/tool-dispatch.ts` | `reference/mcp-tools/*` | **Generated — regenerate, never hand-edit** (Step 3) |
| REST controllers / OpenAPI | `reference/rest-api/*` (rendered from OpenAPI) | Update the YAML: `apps/docs/openapi/time-slack-mcp.yaml` (specs under `specs/*/contracts/openapi.yaml` are committed milestone contracts — extend the docs YAML instead unless told otherwise) |
| Env vars (`apps/api/src/common/config/*.config.ts`, `process.env` reads, `infra/docker/`, compose files) | `reference/environment-variables.mdx` + any `guides/self-hosting/*` page that mentions the var. Also check `.env.example`, `.env.production.example`, and the compose env pass-through (`x-app-env` in `docker-compose.production.yml`) — a var you document as overridable but don't forward there never reaches the containers; fix or flag the gap | Hand-written table: Variable, Default, Required, What it does |
| Docker / compose / deployment | `guides/self-hosting/*` and `tutorials/quickstart.mdx` | Hand-written |
| Slack module (`apps/api/src/modules/slack/`) | `guides/slack/*`, `reference/capture-syntax.mdx` | Hand-written |
| GitHub module, webhooks, email | `guides/integrations/*`, `reference/webhooks-idempotency.mdx` | Hand-written |
| End-user features (work items, views, search, notifications, time tracking, export — `apps/api/src/modules/*`, `apps/web/`) | `guides/users/*` | Hand-written |
| RBAC / permissions | `reference/permissions.mdx` | Hand-written |
| Rate limiting | `reference/rate-limits.mdx` | Hand-written |
| Architecture / tenancy / testing stance | `explanation/*` | Hand-written |
| A roadmap item shipped | Move content out of `guides/roadmap/*` into the right guide; update `reference/feature-status.mdx` | Hand-written |

When a feature ships, check `guides/roadmap/` and `reference/feature-status.mdx`: a page sitting there with a `<ComingSoon />` banner for something that now exists is exactly the drift this skill exists to fix.

## Step 3 — Generated content: regenerate, don't edit

Everything in `apps/docs/content/docs/reference/mcp-tools/` (49 tool pages + `index.mdx` + `meta.json`) is generated by `apps/docs/scripts/generate-mcp-pages.mjs` from the contracts registry. Hand edits there are overwritten on the next build, so they're worse than useless — they hide drift until it reappears.

If the MCP registry, tool input schemas, or `TOOL_PERMISSIONS` changed:

```bash
pnpm --filter @rytask/docs generate:mcp          # regenerate all tool pages
pnpm --filter @rytask/docs check:mcp-docs-drift  # verify pages ↔ registry agree
```

The generator does `require('@rytask/contracts')`, so it needs installed deps and a built contracts package. If it fails to resolve, run `pnpm install` and `pnpm --filter @rytask/contracts build` first.

Commit the regenerated output together with the registry change — CI runs the drift check inside the docs build and refuses to pass if they diverge.

## Step 4 — Edit or add hand-written pages

Read the existing page fully before editing — match its structure and depth, and update *every* statement the change invalidates, not just the obvious one (defaults in tables, example commands, screenshots' descriptions, cross-links from other pages — `grep -r` the content tree for the old name/value).

Conventions the site enforces or assumes:

- **Frontmatter** — every page needs both fields, validated by the Fumadocs page schema:
  ```yaml
  ---
  title: Page title
  description: One sentence that earns its place in search results.
  ---
  ```
- **New pages must be registered** in the sibling `meta.json` `pages` array, in the position they belong in the sidebar — an unlisted page is invisible.
- **Links are root-relative**: `[backups](/docs/guides/self-hosting/backups-and-restore)`.
- **Components** (from `apps/docs/components/mdx.tsx`): `<StatusBadge status="available|in-progress|coming-soon" />`, `<ComingSoon tier="..." />`, `<InProgress />`, plus the standard Fumadocs set (`Callout`, `Tabs`, …).
- **Voice**: plain, kind, jargon-free — it must make sense to a non-technical teammate (the Albert/Marissa test from `knowledge/VISION.md`). Sentence case headings. Explain *why* before *how*. No marketing fluff, no emoji as chrome.
- The env var table format is `| Variable | Default | Required | What it does |`. Defaults must be the literal defaults from the code — copy them from the config source, don't paraphrase.

## Step 5 — Validate

```bash
pnpm --filter @rytask/docs lint        # Biome
pnpm --filter @rytask/docs typecheck   # fumadocs-mdx compile + tsc — catches broken MDX/frontmatter
pnpm --filter @rytask/docs build       # full gate: generate → drift check → next build
```

The build is the gate CI uses; if it's too slow for the iteration loop, run lint + typecheck per edit and the full build once at the end. Fix what they report before declaring the docs updated.

## Step 6 — Report

Summarize for the user: which pages changed and why, which were regenerated vs hand-edited, anything you noticed that's drifted but was out of scope (e.g. a stale roadmap page), and any claim you could not verify against the code (flag it rather than guessing).
